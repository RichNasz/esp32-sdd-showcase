# BLE Beacon + Deep Sleep

## Overview

Duty-cycled BLE beacon example across four supported boards. On every wake cycle the device increments two RTC-persistent counters (`boot_count` and `sequence`), broadcasts non-connectable BLE advertisements for 10 seconds with the user LED on, tears down the BLE stack completely, turns the LED off, and re-enters deep sleep for 10 seconds. The beacon advertises as **`"ESP32-SDD"`** — the name appears directly in BLE scanner apps (e.g. nRF Connect) without any MAC filter. The serial monitor logs the BLE MAC address on each cycle so the device can be located instantly. A button press on the BOOT button triggers an immediate on-demand advertisement via GPIO wakeup (on ESP32-S3 and ESP32 only — see board table). The 50% duty cycle is optimised for easy demonstration.

Board-specific configuration (LED GPIO, button GPIO, console type) is resolved at compile time via Kconfig symbols in `sdkconfig.defaults.<target>`. Select a board with `idf.py set-target`.

**BLE stack choice**: NimBLE is used instead of Bluedroid. NimBLE initialises in ~50 ms (vs. ~200 ms for Bluedroid), consumes ~30 KB RAM (vs. ~60 KB), and provides a clean synchronous `nimble_port_stop()` teardown path — well suited to a duty-cycled beacon followed by immediate deep sleep. See `shared-specs/ble-patterns.md` for the full decision matrix.

> **Serial monitor on USB CDC boards**: On boards that use USB CDC for the serial console (e.g. XIAO ESP32S3), the USB connection drops when the device enters deep sleep. The serial monitor will report an error such as `Device not configured` — this is expected. Serial output is only available while the LED is on (active window). The monitor reconnects automatically when the device wakes on the next cycle.

### Supported Boards

| `idf.py set-target` | Board | LED GPIO | Button GPIO | Button Wakeup | Console |
|---|---|---|---|---|---|
| `esp32s3` | Seeed XIAO ESP32S3 | GPIO 21 (active LOW) | GPIO 0 (RTC_GPIO0) | ext1 | USB CDC |
| `esp32c6` | Seeed XIAO ESP32-C6 | GPIO 15 (active LOW) | GPIO 9 | timer-only | USB Serial/JTAG |
| `esp32c5` | Seeed XIAO ESP32-C5 | GPIO 27 (active LOW) | GPIO 9 | timer-only | USB Serial/JTAG |
| `esp32` | Adafruit HUZZAH32 | GPIO 13 (active HIGH) | GPIO 0 (RTCIO_CH11) | ext0 | UART (CP2104) |

GPIO 9 is the XIAO boot button on C6 and C5, but it is **not an RTC GPIO** on those targets (C6: only GPIO 0-7 are RTC-capable; C5: only GPIO 0-6). Button wakeup is therefore skipped on C6/C5; the device wakes only on the 10-second timer. No external hardware is required.

## Advertising Payload Layout

Total PDU: **25 bytes** (BLE limit: 31 bytes).

| Offset | Length | Field | Value |
|---|---|---|---|
| 0 | 1 | Flags length | `0x02` |
| 1 | 1 | Flags AD type | `0x01` |
| 2 | 1 | Flags value | `0x06` (LE General Discoverable + BR/EDR Not Supported) |
| 3 | 1 | Local Name length | `0x0A` (10 bytes follow) |
| 4 | 1 | Local Name AD type | `0x09` (Complete Local Name) |
| 5 | 9 | Local Name value | `"ESP32-SDD"` |
| 14 | 1 | Manufacturer data length | `0x0B` (11 bytes follow) |
| 15 | 1 | Manufacturer data AD type | `0xFF` |
| 16 | 2 | Company ID (little-endian) | `0xFF 0xFF` (test/example use per Bluetooth SIG) |
| 18 | 2 | `boot_count` (little-endian) | Increments on every wake cycle |
| 20 | 2 | `sequence` (little-endian) | Increments on every wake cycle |
| 22 | 4 | Magic identifier | `0xDE 0xAD 0xBE 0xEF` |

## Power Budget

| State | Current | Duration | Charge / cycle |
|---|---|---|---|
| Active (BLE TX + LED on) | ~30 mA | ~10 s | 300,000 uAs |
| BLE teardown | ~5 mA | ~10 ms | 50 uAs |
| Deep sleep | ~14 uA | ~10 s | 140 uAs |
| **Average (20 s cycle)** | **~15 mA** | -- | 300,190 uAs / 20 s |

The 14 uA sleep floor is the XIAO ESP32S3 board-spec value (see `board-specs/seeed/xiao-esp32s3.md`). The 50% duty cycle is optimised for easy demonstration, not battery life. Reducing the advertising window and increasing the sleep duration would dramatically lower average current.

```
I (213) ble_beacon: Wakeup: EXT1 (button press)
I (213) ble_beacon: Cycle: boot_count=3 seq=3
```

Button press wakeup (ESP32/HUZZAH32 -- GPIO 0 = RTCIO_CH11):

## Key Concepts

- **`RTC_DATA_ATTR`** -- marks variables to survive deep sleep in RTC slow memory; used for `boot_count` and `sequence`. Resets only on power-on or hard reset.
- **Complete Local Name AD** -- `"ESP32-SDD"` is included in every advertising packet so the beacon appears by name in BLE scanner apps. Uses `ble_hs_adv_fields.name` / `.name_len` / `.name_is_complete` in NimBLE.
- **BLE MAC logging** -- `ble_hs_id_copy_addr(BLE_ADDR_PUBLIC, ...)` reads the device public address inside `ble_sync_cb()`. NimBLE stores addresses in little-endian byte order; bytes are printed in reverse for human-readable `XX:XX:XX:XX:XX:XX` format.
- **LED as activity indicator** -- the user LED turns on when advertising starts and off after BLE teardown, before entering deep sleep. This provides a clear visual indicator: LED on = beacon active and serial available; LED off = device sleeping.
- **NimBLE broadcaster-only role** -- observer/central/peripheral roles are disabled in `sdkconfig.defaults`, saving ~4-8 KB RAM.
- **`esp_timer` + semaphore** -- bounds the advertising window to 10 seconds without blocking the NimBLE host task. The timer callback calls `ble_gap_adv_stop()` and gives the semaphore; `app_main` blocks on `xSemaphoreTake`. The task watchdog is unsubscribed during this wait (default WDT timeout is 5 s).
- **BLE teardown sequence** -- `ble_gap_adv_stop()` then `nimble_port_stop()` then `nimble_port_deinit()`. In ESP-IDF 5.5.x, `nimble_port_deinit()` internally handles controller disable and deinit. This exact order is required; `nimble_port_stop()` blocks until the host task exits before the controller is disabled.
- **`gpio_hold_en()` / `gpio_hold_dis()`** -- locks GPIO output state across deep sleep cycles. LED GPIO is held during sleep and released with `gpio_hold_dis()` at the top of each wake cycle.
- **Target-dependent wakeup** -- `esp_sleep_enable_timer_wakeup()` for 10 s cycles on all targets; button-press wakeup uses ext0 on ESP32 (`esp_sleep_enable_ext0_wakeup()`) and ext1 on ESP32-S3 (`esp_sleep_enable_ext1_wakeup(1ULL << BUTTON_GPIO, ESP_EXT1_WAKEUP_ANY_LOW)`). On ESP32-C6/C5 the boot button (GPIO 9) is not an RTC GPIO, so only timer wakeup is configured.
- **Per-target Kconfig** -- `sdkconfig.defaults.<target>` sets `CONFIG_EXAMPLE_LED_GPIO`, `CONFIG_EXAMPLE_LED_ACTIVE_LEVEL`, and the console type. `CONFIG_EXAMPLE_BUTTON_GPIO` is only active on targets that support button wakeup (S3, ESP32). `main.c` reads these symbols so no source changes are needed when switching boards.
- **USB CDC and deep sleep** -- on boards using USB CDC for serial (e.g. XIAO ESP32S3), the USB connection drops when the device enters deep sleep. Serial output is only observable while the LED is on. The monitor reconnects automatically on the next wake cycle.

### Manual Tests -- hardware required

**T-M1 -- Beacon Visible by Name in Scanner**

*Why manual: verifying OTA BLE advertisement content requires a second device acting as a scanner -- not reproducible in simulation.*

Prerequisites: smartphone with nRF Connect (iOS/Android) or equivalent BLE scanner.

1. Flash to XIAO ESP32S3: `idf.py set-target esp32s3 && idf.py build flash`
2. Open the serial monitor -- note the `BLE MAC: XX:XX:XX:XX:XX:XX` line printed on the first wake (LED will be on).
3. Open nRF Connect > Scanner tab > start scanning.
4. Confirm the device appears as **`"ESP32-SDD"`** in the device list (no MAC filter needed). Do not use the "nRF Beacon" filter -- this beacon uses custom manufacturer data, not the iBeacon/Eddystone format.
5. On each device wake (every 10 seconds -- LED turns on), confirm advertisements are visible for 10 seconds.
6. Inspect raw manufacturer data -- confirm company ID `0xFFFF` and 4-byte magic `DE AD BE EF` are present.
7. Confirm `boot_count` field increments by 1 on each wake.

Pass: device listed as "ESP32-SDD" in scanner; advertisement appears on every cycle; manufacturer data matches spec.

---

**T-M2 -- Active Window ~10 s (LED on during broadcast)**

*Why manual: active window timing requires observation of LED on-duration.*

1. Observe the user LED during the advertising window -- it should stay on for approximately 10 seconds.
2. On USB CDC boards (XIAO ESP32S3), the serial monitor disconnects when the LED turns off (device enters deep sleep). This is expected.
3. Serial monitor must show advertising start then BLE teardown approximately 10 seconds apart.

Pass: LED on for ~10 s; total active window <= 12 s per serial timestamps.

---

**T-M3 -- Button Wakeup Triggers Immediate Advertisement (ESP32-S3 and ESP32 only)**

*Why manual: physical GPIO wakeup requires real hardware.*

> **Scope**: Button wakeup is only supported on `esp32s3` and `esp32`. On `esp32c6` and `esp32c5` the boot button (GPIO 9) is not an RTC GPIO, so only timer wakeup is configured; skip this test for those targets.

1. Flash to XIAO ESP32S3 or HUZZAH32.
2. While device is in deep sleep (LED off), press and release the BOOT button (GPIO 0 on both boards).
3. Confirm LED turns on and advertisement burst occurs within 500 ms of button press.
4. Confirm serial log shows `EXT1` wakeup cause (ESP32-S3) or `EXT0` wakeup cause (ESP32).

Pass: advertisement triggered within 500 ms of button press.
